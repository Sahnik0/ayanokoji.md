# Scaffold File Reference

This document describes the four scaffold files generated by the Ayanokoji
strategic execution protocol and the rules governing each.

---

## AGENTS.md

**Purpose:** Master rulebook. Read first every session, before any code.

**Hard constraints:**
- 60-line hard cap. Non-negotiable. If it exceeds 60 lines, compress it.
- Updated only when a fundamental protocol change is made. Never mid-task.
- Must contain: Session Start Protocol (all 6 steps), Hard Rules, Verification Gate, Behavioral Contract.

**The Session Start Protocol** (in order):
1. Read AGENTS.md completely.
2. Read PROGRESS.md — what is done and what has failed.
3. Read TASK.md — current task and acceptance criteria.
4. Read CONTEXT.md — only if the current task touches the modules listed there.
5. Do not write any code until steps 1–4 are complete.
6. Load `skills/ayanokoji/SKILL.md` and apply all Operating Principles before acting.

---

## CONTEXT.md

**Purpose:** Architecture reference. Describes the system — not the file tree.

**Structure:**
- 3–5 sentence project overview (what it does, who uses it, what scale)
- Tech stack: one line per layer (frontend / backend / database / infra / etc.)
- Core modules: each entry has three fields:
  - `purpose:` what this module does
  - `depends-on:` what it relies on
  - `danger zones:` what breaks if you touch it wrong
- Off-limits zones: explicit list cross-referenced with AGENTS.md

**When to update:** Only when the architecture meaningfully changes — new
module, removed layer, changed dependency boundary. Not on every task.

---

## TASK.md

**Purpose:** Current task brief. Completely overwritten per task. Never appended to.

**Required sections:**

```
# Task: <one-line task name>

## What This Task Requires
<concrete description — no vague verbs>

## Files In Scope
- <explicit list of every file this task may touch>

## Files Out Of Scope
- <explicit list — cross-reference AGENTS.md Off-Limits>

## Acceptance Criteria
<runnable check that produces a binary pass/fail>

## Approach
1. <concrete step>
2. <concrete step>
...

## Known Risks
- <risk> → fallback: <what to do if it fails>
```

**Acceptance Criteria rule:** If you cannot run the check and get a clear
pass or fail, it is not an acceptance criterion. Rewrite it until it is.

---

## PROGRESS.md

**Purpose:** Intelligence log. Updated after every completion or failure.

**Three sections only:**

```
## Completed Tasks
| Task | Verified By | Key Files |
|------|------------|-----------|
| ...  | <actual output> | ... |

## Failed Attempts
- <task>: tried <X>, got <exact error>. Must never repeat: <Y>.

## Current Blockers
- <blocker> — needs: <what must happen to unblock>
```

**Rules:**
- Completed Tasks: paste actual verification output, not "tests passed."
- Failed Attempts: exact error + "must never be repeated" clause required.
- Compress entries older than 5 to one-line summaries.
- Clear blockers when resolved — a stale blocker is noise.
